       M4 Implementation Plan: Web Frontend (htmx + PicoCSS)

       ---
       Part 1: Architecture Decision — Hybrid JSON API + Client-Side Templating

       The backend serves a pure JSON REST API. There are no server-side HTML template routes.
        Adding a full server-side HTML fragment rendering layer (the "classic htmx" approach)
       would mean either:
       - Duplicating every API endpoint as an HTML-fragment endpoint, or
       - Rewriting existing routes to content-negotiate between JSON and HTML.

       Both are heavy changes to a stable M1-M3 backend. The practical approach for this
       project:

       Use htmx primarily for navigation/form interception, but render content client-side via
        a small templating layer in app.js. htmx's hx-trigger, hx-on, and event system can
       still drive the UX flow (form submissions, loading indicators, history pushState),
       while a thin render() function in app.js fetches JSON and builds DOM. This is the
       lightest path that avoids a build step, avoids server-side template engines, and keeps
       the backend untouched.

       Socket.io is loaded from the Fastify socket.io server itself (it auto-serves
       /socket.io/socket.io.js), so zero CDN dependency.

       app.setNotFoundHandler((request, reply) => {

       3.2 Socket.io integration in server.js

       Socket.io needs to attach to the Fastify underlying HTTP server. This also requires a
       new file src/ws/socket.js.

       // src/ws/socket.js
       import { Server } from 'socket.io'
       import jwt from 'jsonwebtoken'
       import config from '../config.js'

       let io = null

       export function initSocket(httpServer) {
         io = new Server(httpServer, {
           cors: { origin: config.allowedOrigins }
         })

         io.use((socket, next) => {
           const token = socket.handshake.auth?.token
           if (!token) return next(new Error('AUTH_REQUIRED'))
           try {
             const payload = jwt.verify(token, config.jwtSecret)
             socket.userId = payload.sub
             socket.userRole = payload.role
             next()
           } catch {
             next(new Error('INVALID_TOKEN'))
           }
         })

         io.on('connection', (socket) => {
           socket.on('join_board', ({ board_id }) => {
             socket.join(`board:${board_id}`)
           })
         })

         return io
       }

       export function getIo() { return io }

       New dependency: npm install socket.io

       The AI providers (openai.js, anthropic.js) and agent_api.js need to broadcast WebSocket
        events after writing comments. This means importing getIo() and emitting ai_reply /
       new_comment events. Similarly, board.js POST routes should emit new_post and
       new_comment.

       5.1 Auth & Token Management

       const AUTH = {
         getAccessToken: () => localStorage.getItem('accessToken'),
         getRefreshToken: () => localStorage.getItem('refreshToken'),
         setTokens: (access, refresh) => {
           localStorage.setItem('accessToken', access)
           if (refresh) localStorage.setItem('refreshToken', refresh)
         },
         clear: () => {
           localStorage.removeItem('accessToken')
           localStorage.removeItem('refreshToken')
         },
         isLoggedIn: () => !!localStorage.getItem('accessToken'),
         // Decode JWT payload (no verification — server does that)
         getUser: () => {
           const token = localStorage.getItem('accessToken')
           if (!token) return null
           try {
             return JSON.parse(atob(token.split('.')[1]))
           } catch { return null }
         }
       }

       Token refresh logic: the API client wrapper intercepts 401 responses. If a 401 is
       received and a refreshToken exists, it calls POST /api/auth/refresh, stores the new
       accessToken, and retries the original request once. If refresh also fails, it clears
       tokens and redirects to login.

       5.2 API Client

       async function api(path, options = {}) {
         const headers = { 'Content-Type': 'application/json', ...options.headers }
         const token = AUTH.getAccessToken()
         if (token) headers['Authorization'] = `Bearer ${token}`

         let res = await fetch(`/api${path}`, { ...options, headers })

         // Auto-refresh on 401
         if (res.status === 401 && AUTH.getRefreshToken()) {
           const refreshRes = await fetch('/api/auth/refresh', {
             method: 'POST',
             headers: { 'Content-Type': 'application/json' },
             body: JSON.stringify({ refreshToken: AUTH.getRefreshToken() })
           })
           if (refreshRes.ok) {
             const { accessToken } = await refreshRes.json()
             AUTH.setTokens(accessToken)
             headers['Authorization'] = `Bearer ${accessToken}`
             res = await fetch(`/api${path}`, { ...options, headers })
           } else {
             AUTH.clear()
             navigate('/login')
             throw new Error('Session expired')
           }
         }

         return res
       }

       5.3 Router

       Use pushState based routing for clean URLs. Intercept all <a data-route> clicks.
       Pattern:

       const ROUTES = [
         { pattern: /^\/$/, render: renderBoardList },
         { pattern: /^\/login$/, render: renderLogin },
         { pattern: /^\/register$/, render: renderRegister },
         { pattern: /^\/boards\/(\d+)$/, render: renderPostList },
         { pattern: /^\/posts\/(\d+)$/, render: renderPostDetail },
         { pattern: /^\/profile$/, render: renderProfile },
         { pattern: /^\/agents$/, render: renderMyAgents },
         { pattern: /^\/admin$/, render: renderAdmin },
       ]

       function navigate(path) {
         history.pushState(null, '', path)
         routeTo(path)
       }

       function routeTo(path) {
         for (const route of ROUTES) {
           const match = path.match(route.pattern)
           if (match) {
             route.render(...match.slice(1))
             return
           }
         }
         renderNotFound()
       }

       // Intercept link clicks
       document.addEventListener('click', (e) => {
         const a = e.target.closest('a[data-route]')
         if (a) {
           e.preventDefault()
           navigate(a.getAttribute('href'))
         }
       })

       window.addEventListener('popstate', () => routeTo(location.pathname))

       5.4 Page Renderers

       Each renderer is an async function that fetches data from the JSON API and builds HTML
       into document.getElementById('app'). Key patterns:

       - renderBoardList() — GET /api/boards, render as a <table> or card list
       - renderPostList(boardId) — GET /api/boards/:id/posts, cursor pagination with "Load
       more" button
       - renderPostDetail(postId) — GET /api/posts/:id, renders post + comment tree (2-level
       nesting), includes comment form
       - renderLogin() / renderRegister() — Forms that POST to auth endpoints
       - renderProfile() — GET /api/users/me, editable form
       - renderMyAgents() — GET /api/agents/mine, CRUD forms for agents
       - renderAdmin() — Admin panel with tabs for public agents and user management

       Comment tree rendering (2-level max):

       function renderCommentTree(comments) {
         const topLevel = comments.filter(c => c.parent_id === null)
         return topLevel.map(c => `

       'deleted' : ''}">

       Part 9: Potential Challenges

       1. Markdown rendering: The PRD says posts support Markdown. Without a build step, the
       simplest option is loading marked from CDN (<script
       src="https://cdn.jsdelivr.net/npm/marked/marked.min.js">). This is a single additional
       CDN dependency but keeps things simple. Must sanitize output to prevent XSS (use
       DOMPurify from CDN as well, or strip HTML tags manually).
       2. Agent name display on comments: The current listCommentsByPost query does not join
       agents table. This is a required backend change (Step 1).
       3. System status dashboard (US-17): No backend endpoint exists for this. Options: (a)
       add a simple GET /api/admin/status endpoint that queries job_queue counts and returns
       stats, or (b) defer to M6. Recommend adding a minimal endpoint.
       4. Agent logs (US-13): The GET /api/agents/:id/logs route was noted as "omitted" in M3
       progress (YAGNI). For the frontend to show logs, this route needs to be added. It's a
       simple route wrapping the existing listAgentLogs query function.
       5. Socket.io reconnection on token refresh: When the access token is refreshed, the
       socket connection uses the old token. Need to disconnect and reconnect with the new
       token after a refresh.
