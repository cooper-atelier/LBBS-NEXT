# M1 实施计划：DB Schema + Auth + 基础 CRUD

## Context

lbbs-next 是一个全新项目（greenfield），目前只有文档文件（CLAUDE.md、PRD.md、AGENTPLAN.md），无任何源代码。M1 的目标是让多用户注册、登录、发帖、回帖完整跑通，为后续 M2（AI 系统）打好基础。

建表时一次性创建全部 7 张表（包括 agents、job_queue、agent_logs），避免 M2 需要做 migration。

## 各文件关键设计

### Step 1: package.json + 项目配置

`package.json`:
- `"type": "module"` (ES Modules)
- `"bin": { "lbbs-next": "./bin/cli.js" }`
- 依赖: `fastify`, `@fastify/cors`, `@fastify/rate-limit`, `better-sqlite3`, `bcrypt`, `jsonwebtoken`, `dotenv`
- 开发依赖: `vitest`, `nodemon`
- scripts: `dev` (nodemon), `dev:init` (node bin/cli.js init), `start`, `test` (vitest run)

`.env.example`: 包含 PORT, HOST, DATA_DIR, JWT_SECRET, JWT_REFRESH_SECRET, ENCRYPTION_KEY, ALLOWED_ORIGINS, LOG_LEVEL

`.gitignore`: node_modules/, data/, .env, *.db

### Step 2: src/config.js

- 项目中唯一 import dotenv 的文件，唯一读 process.env 的文件
- 启动时校验必填项: JWT_SECRET, JWT_REFRESH_SECRET, ENCRYPTION_KEY
- ENCRYPTION_KEY 校验长度必须为 64 hex 字符（32 字节）
- 导出 `Object.freeze(config)` 防止意外修改
- 包含硬编码常量: BCRYPT_ROUNDS=12, JWT_ACCESS_EXPIRES='15m', JWT_REFRESH_EXPIRES='7d', SYSTEM_USER_ID=1, RATE_LIMIT_GLOBAL=100, RATE_LIMIT_LOGIN=5

### Step 3: src/utils/crypto.js

- `encrypt(plaintext)` / `decrypt(stored)` — AES-256-GCM，格式 `iv:tag:ciphertext`（全 hex）
- `sha256(input)` — Agent token 哈希存储用
- `hmacSha256(data, key)` — Webhook 签名用（M2 使用，M1 先实现）
- 密钥从 config.encryptionKey 读取

### Step 4: src/db/migrations/001_init.sql

- 一次性创建全部 7 张表，顺序: users → boards → posts → comments → agents → job_queue → agent_logs
- 全部使用 `CREATE TABLE IF NOT EXISTS`（幂等）
- Schema 完全照搬 PRD §4.1
- 包含 `CREATE UNIQUE INDEX idx_public_agent_name ON agents(name) WHERE owner_id IS NULL`
- 包含系统用户种子: `INSERT OR IGNORE INTO users (id, ...) VALUES (1, 'system', ...)`

### Step 5: src/db/init.js

- 导出 `getDb()` 单例 — 自动创建 data/ 目录，打开 bbs.db
- 每次连接设置 `pragma('foreign_keys = ON')` 和 `pragma('journal_mode = WAL')`
- 导出 `initializeDatabase()` — 读取并执行 001_init.sql

### Step 6: src/db/queries.js

项目中唯一包含 SQL 的文件。按领域组织:

**Users**: createUser, findUserByUsername, findUserByEmail, findUserById（公开，不含 password_hash）, findUserByIdInternal（含 token_version，JWT 验证用）, updateUserProfile, incrementTokenVersion

**Boards**: createBoard, listBoards, findBoardById

**Posts**: createPost, listPostsByBoard（cursor-based 分页）, findPostById（JOIN author 信息）, updatePost, softDeletePost, findPostOwner

**Comments**: createComment, listCommentsByPost（返回全部含软删除，保留树结构）, findCommentById, updateComment, softDeleteComment, findCommentOwner

**工具函数**: `serializeTimestamps(row)` — 将 Unix 秒转为 ISO 8601

cursor-based 分页: `WHERE p.board_id = ? AND p.is_deleted = 0 AND (? IS NULL OR p.id < ?) ORDER BY p.id DESC LIMIT ?`，返回 `{ posts, nextCursor }`

### Step 7: src/middleware/auth.js

- `verifyJWT(request, reply)` — 解析 Bearer token，验证签名 + token_version + is_active，设置 request.user
- `requireAdmin(request, reply)` — 检查 request.user.role === 'admin'，必须在 verifyJWT 之后使用

### Step 8: src/middleware/rateLimit.js

- `registerRateLimit(fastify)` — 注册全局限流 100req/min/IP
- 导出 `loginRateLimit` 配置对象 — 登录路由单独 5req/min/IP

### Step 9: src/middleware/validate.js

- `ownerOrAdmin(request, reply, resourceOwnerId)` — 检查当前用户是资源所有者或管理员

### Step 10: src/api/auth.js

4 个路由，每个都有完整 Fastify schema:

| 路由 | 关键逻辑 |
|------|---------|
| POST /register | 查重 username/email → bcrypt(12) → INSERT → 返回 201（不返回 token） |
| POST /login | findByUsername → bcrypt.compare → 签发 accessToken(15m) + refreshToken(7d)，payload 含 token_version |
| POST /refresh | 用 JWT_REFRESH_SECRET 验证 → 比对 DB token_version → 签发新 accessToken |
| POST /logout | preHandler: verifyJWT → incrementTokenVersion → 所有旧 token 失效 |

### Step 11: src/api/users.js

| 路由 | 说明 |
|------|------|
| GET /me | verifyJWT → 返回当前用户完整资料（不含 password_hash） |
| PATCH /me | verifyJWT → 只允许更新 avatar_url, bio |
| GET /:id | 公开路由 → 返回 id, username, avatar_url, bio, created_at |

### Step 12: src/api/board.js

M1 最大的文件，处理 boards + posts + comments:

**Boards**: GET /boards（公开）, POST /boards（admin only）

**Posts**:
- GET /boards/:id/posts — cursor 分页，过滤 is_deleted=0
- POST /boards/:id/posts — verifyJWT，返回 201（不触发 AI，M2 在此处集成 detector）
- GET /posts/:id — 详情 + 全部评论（软删除的替换内容为 `[此内容已删除]`）
- PATCH /posts/:id — owner only，不触发 AI（架构决策 #9）
- DELETE /posts/:id — owner/admin，软删除

**Comments**:
- POST /posts/:id/comments — verifyJWT，支持 parent_id（2 层嵌套限制：parent 的 parent_id 必须为 null）
- PATCH /comments/:id — owner only
- DELETE /comments/:id — owner/admin，软删除

### Step 13: src/server.js

- `buildApp()` — 创建 Fastify 实例，注册插件和路由，返回 app（测试用）
- `startServer()` — initializeDatabase() + buildApp() + listen + 优雅关闭
- 插件注册顺序: CORS → rate-limit → auth routes → user routes → board routes
- 全局错误处理: validation error → 400, 其他 → 500，统一 `{ error, message }` 格式

### Step 14: bin/cli.js

- `#!/usr/bin/env node`
- 支持 `init`（建表）和 `start`（启动服务）两个命令
- 使用动态 import() 按需加载，init 不加载完整 server

## 测试策略

框架: vitest（ESM 原生支持）

使用 `buildApp()` + Fastify `app.inject()` 做 HTTP 测试，不需要真实端口。每个测试文件使用临时数据库。

关键测试用例:
- auth: 注册/登录/刷新/登出/token_version 吊销/系统用户无法登录
- board: admin 建板/发帖/cursor 分页/软删除列表隐藏/软删除详情内容替换/2 层评论嵌套限制/owner 权限校验
- crypto: encrypt/decrypt 往返测试

## 验证方式

```bash
# 初始化 + 启动
node bin/cli.js init
node bin/cli.js start &

# 注册
curl -X POST localhost:3000/api/auth/register \
  -H 'Content-Type: application/json' \
  -d '{"username":"alice","email":"alice@test.com","password":"password123"}'
# → 201

# 登录
curl -X POST localhost:3000/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"alice","password":"password123"}'
# → 200 { accessToken, refreshToken }

# 创建板块（需要 admin）
# 先用 CLI 或直接 DB 创建 admin，然后:
curl -X POST localhost:3000/api/boards \
  -H 'Authorization: Bearer <admin_token>' \
  -H 'Content-Type: application/json' \
  -d '{"name":"技术讨论","description":"聊技术"}'
# → 201

# 发帖
curl -X POST localhost:3000/api/boards/1/posts \
  -H 'Authorization: Bearer <token>' \
  -H 'Content-Type: application/json' \
  -d '{"title":"第一帖","content":"Hello World"}'
# → 201

# 回帖
curl -X POST localhost:3000/api/posts/1/comments \
  -H 'Authorization: Bearer <token>' \
  -H 'Content-Type: application/json' \
  -d '{"content":"好帖！"}'
# → 201

# 运行测试
npm test
```
